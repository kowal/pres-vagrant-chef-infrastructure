Vagrant & Chef

Infrastructure as a code

Why?

  • People work faster
  • Don't focus on configuration but on the PRODUCT!
  • Improves Engenering, QA, Demo and CI

Local install issues

  • everything must be installed manually
  • developers don't know the "production" config
  • a lot of “works on my machine” bugs
  • multiple projects
  • keep development environment in sync with the team
  • keep wiki updated
  • nobody reads the wiki anyway
  • long onboarding time for new team members
  • developers are forced to one particular OS

Vagrant

"tool for creating and distributing virtualized development environments"

The Vagrant Way

  • encourages automation to set up environment
  • simple workflow 
    
					$ git checkout ...
					$ vagrant up
					# code, code, code ...
					$ git pull
					$ vagrant provision

The Vagrant Way

  • developers uses own editors, browsers, and tools
  • existence of Vagrant is transparent and unimportant
  • operations engineers work on system automation scripts
  • they test those scripts on Vagrant boxes
  • production == development

Vagrant provisioners

  • puppet
  • chef
  • shell scripts

Chef

"systems and cloud infrastructure automation framework that makes it easy to deploy servers and applications to any physical, virtual, or cloud location"

Why Chef?

  • "configuration management as a service"
  • "remove people from the process of building automated infrastructure"
  • "powered by Ruby and a simple DSL"

How it works?

  • nodes, roles
  • cookbooks, recipes, resources
  • run list

Chef Modes

  • cookbooks in central repo
  • cookbooks on the node (aka chef-solo)
